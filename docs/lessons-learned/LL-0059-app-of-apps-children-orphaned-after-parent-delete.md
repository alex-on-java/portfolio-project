# LL-0059: Deleting the Parent of an App-of-Apps Strands the Whole Subtree

## Summary

In an App-of-Apps hierarchy, a parent `Application` manages its children (other `Application` and `ApplicationSet` objects) the same way any `Application` manages any workload: by the `app.kubernetes.io/instance` label, not by `metadata.ownerReferences`. When `kubectl delete application/<parent>` runs against a parent that carries no `resources-finalizer.argocd.argoproj.io`, the apiserver removes the parent CR instantly. Each child `Application` still carries its own finalizer, but no controller is now driving its deletion: the finalizer is dormant by itself, not self-executing. The Kubernetes garbage collector finds nothing to collect because no managed object (neither the child `Application`s nor their workloads) points back at the parent via `ownerReferences`. Its CR vanishes; the children and their workloads remain.

## What Happened

The destroy path in `infra/bootstrap/main.tf` once provisioned the bootstrap `Application` without a cascade finalizer. A teardown call of `kubectl delete application/bootstrap --cascade=foreground -n argocd` returned promptly; `kubectl wait --for=delete` likewise returned green within seconds. Every `Application` generated by the bootstrap `ApplicationSet`s (`contour`, `kargo`, `cert-manager`, and siblings) stayed in the cluster, each carrying its own `resources-finalizer.argocd.argoproj.io` from its `ApplicationSet` template, and each holding live `Deployment`, `Service`, and `Namespace` objects. The subsequent terraform destroy of the network module failed against a `k8s-<cluster-uid>-node-http-hc` firewall that the orphaned `contour` LoadBalancer had caused the GKE cloud-controller-manager to provision and never reclaim.

The fix in this branch restores the cascade finalizer on the bootstrap `Application` unconditionally so the same `kubectl delete` drives the controller-mediated cascade documented in [LL-0052](LL-0052-kubectl-cascade-foreground-leaves-argocd-workloads-behind.md).

## Root Cause

A finalizer is a string in `metadata.finalizers`; it is not a controller. The `resources-finalizer.argocd.argoproj.io` value tells the `argocd-application-controller` that, when an `Application` enters a deleting state, it must walk that `Application`'s managed live tree before allowing the CR to vanish. It is acted on only when the controller observes a `deletionTimestamp` on the object that carries it. Nothing inside a finalizer makes a controller find the object; only an external delete attempt does.

An App-of-Apps hierarchy compounds this. The parent `Application` references a directory of `ApplicationSet` manifests under its `source.path`; the application controller reconciles those manifests into the cluster and labels each `ApplicationSet` with `app.kubernetes.io/instance=<parent>`. The `ApplicationSet` controller then generates `Application` objects, and these do carry `metadata.ownerReferences` back to the `ApplicationSet`. That single ownerReference link is the only Kubernetes-GC-honoured edge in the tree, and it points sideways, not upward. It would fire if an `ApplicationSet` were deleted, but the parent `Application` is not an owner of any `ApplicationSet` it manages. Workloads sit one level deeper still and carry no ownerReference to any ArgoCD object at all.

When `kubectl delete` removes the parent CR with no finalizer present, the chain breaks at the top. No controller is reconciling the parent's desired-state tree any more, so no actor sees the orphaned `ApplicationSet`s as drift to delete. The children's finalizers are still present, still naming the same controller, and still inert because no one is driving the children toward deletion. Kubernetes GC inspects `ownerReferences` and finds nothing pointing at the deleted parent. Every layer below the parent stays exactly as it was.

The reverse experiment closes the loop. Adding `resources-finalizer.argocd.argoproj.io` to the parent before the same `kubectl delete` enrols the application controller. The controller drains the parent's managed tree object-by-object: it deletes each `ApplicationSet`, and the `ApplicationSet` controller follows its own ownerReference into the generated child `Application`s. The application controller picks up each of those (now in a deleting state with finalizer present) and drains their workloads. Only then does each finalizer clear and each object vanish.

## Distinction from LL-0005 and LL-0052

[LL-0005](LL-0005-application-deletion-hangs-despite-preserve-resources.md) and [LL-0052](LL-0052-kubectl-cascade-foreground-leaves-argocd-workloads-behind.md) both frame the single-`Application` case. LL-0005 covers the over-suppression that hangs deletion against a torn-down target cluster. LL-0052 covers a flat hierarchy where `kubectl --cascade=foreground` cannot substitute for an ArgoCD cascade because workloads carry no ownerReference back to their `Application`. This entry is the nested-hierarchy consequence. The misconception it corrects is not "kubectl `--cascade` substitutes for ArgoCD cascade" (LL-0052's framing) but the subtler "child finalizers will catch the cascade even if the parent skips it". They will not. A finalizer cannot trigger itself, and no path from a vanished parent reaches the children that hold one.

## How to Detect

A parent `Application` deleted by `kubectl delete` returning in under five seconds against a non-trivial tree is the canonical signature. After the call, the following all hold simultaneously:

```bash
kubectl get application/<parent> -n argocd
# Error from server (NotFound)

kubectl get applicationset -n argocd
# child ApplicationSets still present, each labelled app.kubernetes.io/instance=<parent>

kubectl get application -n argocd
# child Applications still present, each with metadata.finalizers including
# resources-finalizer.argocd.argoproj.io

kubectl get -A -l app.kubernetes.io/instance=<child>
# Deployments, Services, Namespaces still Active under each child
```

The finalizers on the children are the diagnostic tell: they look like a cascade in progress, but their `deletionTimestamp` is empty. Nothing has asked the children to delete, so nothing about their state will change. The tree is stable in the orphaned shape, not transiting through it.

## Adoption Rule

For any `Application` that owns other ArgoCD objects (an App-of-Apps parent, a top-level bootstrap, or any node in a deeper hierarchy), the cascade finalizer must be present on that object before any `kubectl delete` reaches it. Verify with `kubectl get application/<name> -n argocd -o jsonpath='{.metadata.finalizers}'`; if the field is empty, neither the kubectl propagation policy nor finalizers further down the tree will rescue the teardown. Both levers from LL-0052 apply identically here: a declarative finalizer in Git, or `argocd app delete --cascade` patching it in at delete time. The nesting does not change which lever is needed, only the cost of skipping it.
